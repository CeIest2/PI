**Graph Schema:** {schema}
**Research Context:** {research_context}

{history_text}

### MISSION
Analyze the HISTORY of attempts to fulfill the User Intent. Decide if the query is **VALID**, needs a logical **CORRECTED** fix, or if you need to conduct **RESEARCH** to unblock the situation.

### AUDIT STEPS (Mandatory)
1. **VALIDATION**: Check if the query returned 0 rows. Usually, 0 rows mean the query is invalid, UNLESS the request explicitly aims to prove the absence or non-existence of something.
2. **STATISTICAL SKEPTICISM (Data Incoherence)**: If the results are suspiciously uniform (e.g., 50 rows all showing exactly 0.0, null, or false), **DO NOT VALIDATE**. This typically indicates that the property exists but is not populated. Mark as **RESEARCH**.
3. **TECHNICAL SKEPTICISM**: If you suspect a property name or a label is incorrect based on the schema, do not guess. Request **RESEARCH**.
4. **CONTEXTUAL SKEPTICISM**: If you do not know the real-world link between two entities, request **RESEARCH**.

### DECISION STRATEGY (status)
- **"VALID"**: The query is logically sound AND the results are coherent and meaningful.
- **"CORRECTED"**: You have a specific Cypher fix based on the schema or previous research.
- **"RESEARCH"**: You need to investigate (schema check, data quality check, or entity discovery).

### CRITICAL RULE: HANDLING INCOHERENT DATA
If the database returns success but with values that make no statistical sense (e.g., 0.0 everywhere), declare the request unfeasible or request discovery. **Better to state "Data not found" than to interpret 100 rows of 0.0.**

### OUTPUT FORMAT (JSON)
{{
    "status": "VALID" | "CORRECTED" | "RESEARCH",
    "explanation": "Why this decision? Reference specific evidence.",
    "correction": "Cypher query (for CORRECTED) OR specific Natural Language description of investigation (for RESEARCH) OR null (for VALID)"
}}
